# Plan Command - 融合 Superpowers 理念

## 核心理念

**设计先行 (Design-First)**: 任何创造性工作必须先理解需求、设计方案、获得批准，然后才能进入计划阶段。

**HARD-GATE**: 设计文档未获用户批准，禁止进入计划阶段，禁止任何代码生成。

## 执行流程

### Phase 1: 检查设计规格

**如果用户已提供设计规格文档（docs/specs/*-design.md）：**
- 读取设计规格
- 验证设计规格是否完整
- 进入 Phase 2

**如果用户没有设计规格：**
- 提示用户需要先进行设计探索
- 引导用户使用 `/brainstorm` 命令
- **在此阶段禁止生成任何计划**

### Phase 2: 调用 planning skill 生成计划

**激活 planning skill**: 使用 skill 工具加载 planning skill

planning skill 会：

1. **验证设计规格**
   - 检查是否覆盖所有需求
   - 检查是否包含验收标准
   - 如有问题，报告给用户

2. **文件结构规划**
   - 映射需要创建/修改的文件
   - 每个文件明确职责
   - 遵循项目现有模式

3. **任务分解**
   - 每个任务 2-5 分钟可完成
   - 包含测试步骤
   - 包含提交步骤
   - 使用 TDD 流程
   - 遵循 DRY、YAGNI 原则

4. **生成计划文档**
   - 路径: `docs/plans/YYYY-MM-DD-<feature>-plan.md`
   - 每个任务包含完整代码示例
   - 明确文件路径
   - 明确执行命令

5. **计划审查循环**
   - 内部审查计划完整性
   - 如有问题，修复后重新审查
   - 最多5轮

### Phase 3: 输出结果

planning skill 执行完成后：

1. **显示计划摘要**
   - 任务数量
   - 预计复杂度
   - 关键风险

2. **等待用户确认**
   - "计划已生成，是否执行？"
   - 如需修改，说明要改什么

---

## 任务执行方式

计划生成后，用户可以选择：

1. **使用 /execute 命令逐个执行任务**
   - 两阶段审查（规格合规 → 代码质量）
   - 每个任务完成后需要审查

2. **直接手动执行**
   - 如果用户选择跳过自动化

---

## 禁止事项

| 反模式 | 正确做法 |
|--------|----------|
| 没有设计规格就生成计划 | 先用 /brainstorm 完成设计 |
| 计划中包含"写代码"而非具体代码 | 计划必须包含完整代码示例 |
| 跳过 TDD 步骤 | 每个任务必须包含测试步骤 |
| 忽略 DRY/YAGNI 原则 | 计划必须遵循这些原则 |

---

## 调用 planning skill

**使用 skill 工具**:
```
skill: planning
```

**传递给 planning skill 的信息**:
- 设计规格文档路径
- 用户需求描述
- 项目上下文
