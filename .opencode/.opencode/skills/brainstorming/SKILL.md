# Brainstorming - 设计探索流程

## 概述

帮助用户将模糊的想法转化为明确的设计规格。通过自然对话协作，理解需求、探索方案、形成设计文档。

**这是所有创造性工作的起点**。任何新功能、新特性、修改行为的请求，都必须先经过此流程。

## 核心原则

### HARD-GATE（硬性约束）
**在设计文档被用户明确批准前，禁止任何代码生成、文件修改、脚手架操作。**

无论任务看起来多简单——一个函数、一个配置、一个样式调整——都必须经过完整设计流程。

### 反模式警告

| 错误想法 | 正确做法 |
|----------|----------|
| "这很简单，不需要设计" | 简单项目设计可以很短，但必须有 |
| "我先看看代码再设计" | 先设计，再验证与代码的兼容性 |
| "用户没说要设计文档" | 设计文档是内部过程产物 |
| "我记得需求，直接开始" | 必须经过澄清问题阶段 |

## 流程步骤

### Step 1: 探索项目上下文

- 读取 README、现有代码结构、相关文档
- 了解当前技术栈和约束
- 检查是否有相关现有代码可参考

### Step 2: 澄清性问题（一次一个）

**规则**: 一次只问一个问题，等待回答后再问下一个

示例问题：
- "这个功能的**目标**是什么？解决什么问题？"
- "有什么**技术约束**？（语言、框架、性能要求）"
- "成功标准是什么？怎样算**完成**？"
- "有没有**现有代码**可以作为参考？"
- "有什么**用户体验**要求？"

### Step 3: 方案探索（2-3个方案）

对于每个方案说明：
- **优点**: 为什么考虑这个方案
- **缺点**: 潜在问题
- **推荐**: 是否推荐

格式：
```markdown
### 方案 A: [名称]
- 优点: ...
- 缺点: ...
- 推荐度: ⭐⭐⭐

### 方案 B: [名称]
- 优点: ...
- 缺点: ...
- 推荐度: ⭐⭐

### 推荐: 方案 [X]
理由: ...
```

### Step 4: 呈现设计（分段批准）

**规则**: 设计分多次呈现，每次等待用户确认后再继续

分段建议（根据项目复杂度调整）：
1. 架构概述（1-2句话 + 简易图）
2. 核心组件设计
3. 数据流 / API 设计
4. 错误处理策略
5. 测试策略

**每段后询问**: "这部分设计是否正确？确认后我继续呈现下一部分。"

### Step 5: 写入设计文档

路径: `docs/specs/YYYY-MM-DD-<feature>-design.md`

文档结构：
```markdown
# [功能名称] 设计规格

## 1. 背景与目标
[为什么做，做什么]

## 2. 技术方案
[选定的方案及理由]

## 3. 架构设计
[组件图、数据流]

## 4. 接口设计
[API 定义]

## 5. 错误处理
[异常情况处理]

## 6. 测试策略
[测试方法]

## 7. 风险与约束
[已知风险]

## 8. 验收标准
[如何验证功能完成]
```

### Step 6: 设计审查

**可选**: 如果设计复杂，可以调用其他 agent 进行审查

检查清单：
- 完整性：是否覆盖所有需求？
- 一致性：是否与现有系统兼容？
- 可行性：技术方案是否可实现？
- 清晰性：描述是否无歧义？

### Step 7: 用户最终确认

> "设计文档已写入 `docs/specs/2026-03-13-auth-design.md`，请审查并确认。确认后我将进入计划阶段。"

**必须等待用户明确批准（如 "可以"、"确认"、"开始"）后再继续**

```

═══════════════════════════════════════════════════════════

   用户希望：...

❓ 澄清问题（待回答）
   Q1: ...
   Q2: ...

❓ 澄清问题（已回答）

   Q2: 答案

📐 方案对比
   方案A: ...
   方案B: ...

📝 设计概要（第一部分）
   [内容]
   
   [请确认这部分设计是否正确，确认后我继续呈现剩余部分]
```

```

                    设计探索 - 已批准 ✅

═══════════════════════════════════════════════════════════

下一步：使用 /plan 命令生成实现计划
```

## 与其他技能的关系

- **planning skill**: brainstorming 完成后，调用 planning skill 生成实现计划
- **tdd skill**: 计划执行阶段使用 tdd skill
- **code-review skill**: 代码审查阶段使用 code-review skill

## 注意事项

1. **不要跳过任何步骤**: 即使是简单的任务，也要完整走流程
2. **一次只问一个问题**: 避免一次抛出多个问题
3. **用户批准后才能继续**: 设计未批准绝对不能进入计划阶段
4. **设计文档要具体**: 避免模糊描述，提供具体方案
5. **考虑边界情况**: 在设计阶段考虑错误处理、边界条件
