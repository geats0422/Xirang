# Brainstorm Command - 设计探索流程

## 核心理念

**设计先行**: 任何创造性工作（新建功能、添加特性、修改行为）都必须先经过设计探索流程。

**HARD-GATE**: 设计文档未获用户批准，禁止进入计划阶段，禁止任何代码生成。

## 执行流程

### Phase 1: 激活 brainstorming skill

**加载 skill**: 使用 skill 工具加载 brainstorming skill

```
skill: brainstorming
```

### Phase 2: 执行设计探索

brainstorming skill 会引导完整的对话式设计流程：

1. **探索项目上下文**
   - 读取 README、现有代码结构、相关文档

2. **澄清性问题**
   - 一次只问一个问题
   - 理解目的、约束、成功标准

3. **方案对比**
   - 提出 2-3 个方案
   - 说明优缺点和推荐

4. **呈现设计**
   - 分段呈现
   - 每段等待用户确认

5. **写入设计文档**
   - 路径: `docs/specs/YYYY-MM-DD-<feature>-design.md`

6. **用户最终确认**
   - 等待用户明确批准

### Phase 3: 后续行动

设计获批后：

1. **提示用户进入计划阶段**
   - "设计已批准，可以使用 /plan 命令生成实现计划"

2. **或者直接生成计划**（如果用户要求）
   - 调用 /plan 命令

---

## 使用方式

```
/brainstorm              → 开始新的设计探索
/brainstorm 用户认证     → 直接指定探索主题
```

---

## 禁止事项

| 反模式 | 正确做法 |
|--------|----------|
| "这很简单，不需要设计" | 必须经过 brainstorming |
| 直接开始写代码 | 设计获批后才能进入计划 |
| 跳过澄清问题 | 充分理解需求后再设计 |

## 与其他命令的关系

| 命令 | 说明 |
|------|------|
| `/brainstorm` | 设计探索 → 设计文档 |
| `/plan` | 设计文档 → 实现计划 |
| `/execute` | 计划 → 执行 + 审查 |
| `/tdd` | 直接 TDD 开发（不推荐，跳过设计） |

**推荐流程**: `/brainstorm` → `/plan` → `/execute`
