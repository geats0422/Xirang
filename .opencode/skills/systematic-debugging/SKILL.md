# Systematic Debugging - 系统化调试

## 核心原则

**铁律**: 在尝试任何修复之前，必须先找到根本原因。症状修复是失败。

```
没有根本原因调查，禁止修复
```

## 适用场景

- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别要使用的情况**:
- 时间紧迫时（紧急情况更容易猜测）
- "就一个快速修复"看起来很明显时
- 已经尝试过多个修复
- 之前的修复没起作用
- 不完全理解问题时

## 调试流程

### Phase 1: 根本原因调查

**在尝试任何修复之前:**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 通常包含确切解决方案
   - 完全阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致地复现**
   - 能可靠地触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变更可能导致这个问题？
   - Git diff、最近的提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **在多组件系统（CI → build → signing, API → service → database）中：**
   
   **在提出修复之前，添加诊断工具:**
   ```
   对于每个组件边界:
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每层的状态
   
   运行一次收集证据显示在哪里出问题
     然后分析证据确定失败的组件
     然后调查该特定组件
   ```

### Phase 2: 验证假设

**基于收集的证据:**
- 形成假设
- 设计验证实验
- 验证/否定假设

### Phase 3: 实施修复

**仅在确定根本原因后:**
- 实施最小修复
- 验证修复有效
- 确保没有引入新问题

### Phase 4: 总结教训

- 记录发现了什么
- 记录为什么之前的方法失败了
- 记录修复如何解决问题

## 禁止事项

| 反模式 | 正确做法 |
|--------|----------|
| 直接打补丁 | 必须先找到根本原因 |
| 跳过错误信息 | 仔细阅读所有错误和警告 |
| 猜测修复 | 基于证据形成假设 |
| "这很简单" | 简单 bug 也有根本原因 |
