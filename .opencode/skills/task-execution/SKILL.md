# Task Execution - 任务执行 + 两阶段审查

## 概述

执行计划中的单个任务，包含实现和两阶段审查。

## 核心原则

### 上下文隔离
- 每个子 agent 只获得任务所需的精确上下文
- 不继承完整会话历史
- 协调者负责构造和传递上下文

### 两阶段审查
- 阶段A: 规格合规审查（是否符合设计？）
- 阶段B: 代码质量审查（代码是否达标？）

## 流程步骤

### Step 1: 准备隔离上下文

从计划文档提取：
- 任务描述
- 涉及的文件
- 验收标准
- 相关的设计规格

**禁止**: 传递完整的会话历史给子 agent

### Step 2: 派发实现 Agent

根据任务类型选择 agent：
- 机械实现 (1-2文件) → 使用低成本模型
- 集成任务 → 使用标准模型
- 架构决策 → 使用最强模型

传递给实现 agent 的信息：
```
## 任务: [任务描述]

## 设计规格: [相关设计文档路径]

## 需要创建/修改的文件:
- [文件列表]

## 验收标准:
- [标准1]
- [标准2]

## 上下文:
[仅包含任务所需的代码片段和文档]
```

### Step 3: 实现 Agent 工作

实现 agent 执行：
1. 实现代码
2. 编写测试
3. 运行测试
4. 自我审查
5. 提交

返回状态：
- **DONE**: 完成，进入审查
- **DONE_WITH_CONCERNS**: 完成但有顾虑，需评估
- **NEEDS_CONTEXT**: 需要补充信息
- **BLOCKED**: 被阻塞，需升级处理

### Step 4: 阶段A - 规格合规审查

**调用**: spec-compliance-reviewer agent 或手动审查

检查清单：
- [ ] 是否实现了设计文档中的所有要求？
- [ ] 是否有遗漏的功能点？
- [ ] 是否有超出设计规格的额外实现？（YAGNI 检查）
- [ ] 接口签名是否符合设计？

**结果**:
- 通过 → 进入阶段B
- 不通过 → 返回 Step 2 修复

### Step 5: 阶段B - 代码质量审查

**调用**: code-reviewer agent

检查清单：
- [ ] 代码风格符合项目规范？
- [ ] 测试覆盖充分？
- [ ] 无安全漏洞？
- [ ] 无性能问题？

**结果**:
- 通过 → 标记任务完成
- 不通过 → 返回 Step 2 修复

### Step 6: 更新进度

- TodoWrite 标记完成
- 汇报给用户
- 询问是否继续下一个任务

## Agent 协作示例

```
用户: /execute 1

→ [读取 docs/plans/2026-03-13-auth-plan.md]
→ [提取任务1: 实现 TokenService]
→ [构造隔离上下文: TokenService 需求 + 相关代码]

→ 派发 @build (MiniMax-M2.5)
   - 实现 TokenService
   - 编写测试，5/5 通过
   - 自我审查：发现可优化点，已改进
   - 提交

→ 阶段A: 规格合规审查 (@spec-reviewer)
   - 检查: 是否符合 design.md 中 TokenService 规格？
   - 结果: ✅ 通过

→ 阶段B: 代码质量审查 (@code-reviewer)
   - 检查: 风格、测试、安全
   - 结果: ✅ 通过

→ [标记任务1完成]
→ 汇报: 任务1完成，耗时X分钟，准备执行任务2？
```

## 审查结果处理

### 阶段A 不通过

```
❌ 阶段A: 规格合规审查 - 不通过

问题:
1. [严重]: 设计中要求支持 Refresh Token，但实现未包含
2. [一般]: 错误码定义与设计不一致

→ 返回实现阶段修复
```

### 阶段B 不通过

```
❌ 阶段B: 代码质量审查 - 不通过

问题:
1. [严重]: 缺少输入验证
2. [中等]: 测试覆盖率仅 60%

→ 返回实现阶段修复
```

## 模型选择策略

| 任务复杂度 | 特征 | 推荐模型 |
|-----------|------|----------|
| 机械实现 | 1-2文件，规格清晰 | MiniMax-M2.5 (快/便宜) |
| 集成任务 | 多文件协调，需判断 | GPT-5.3 (标准) |
| 架构任务 | 设计决策，广泛理解 | Gemini-3 (最强) |

## 错误处理

### 实现 agent 返回 BLOCKED

1. 分析阻塞原因
2. 如是上下文问题 → 补充上下文，重新派发
3. 如是任务太大 → 拆分成更小的任务
4. 如是计划问题 → 上报用户

### 审查发现严重问题

- 必须修复后才能继续
- 记录问题，供用户决策

## 注意事项

1. **隔离上下文**: 不要传递完整历史
2. **两阶段审查**: 都不能跳过
3. **精确匹配**: 只实现设计规格中的内容
4. **及时反馈**: 发现问题立即上报
