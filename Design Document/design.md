📝 **设计概要（第四部分：测试策略 + 风险与约束 + 验收标准）**

**1. 测试策略**

- **单元测试**
  - 认证：密码哈希、JWT 签发/校验、refresh rotation、会话吊销
  - 文档摄入：任务状态机、文件类型/大小校验、失败重试
  - PageIndex 适配层：树索引生成调用、检索结果标准化、错误降级
  - 题库模块：基础题生成、章节归属、难度分层、题型分布
  - run 规则：三种模式的组题器、计分器、奖励倍率、结算器
  - 错题模块：错题记录、embedding 任务触发、相似错题召回
  - LLM/provider 抽象：默认模型调用、结构化输出解析、失败兜底
- **集成测试**
  - 注册/登录/refresh/logout 全流程
  - 上传文档 -> 后台处理 -> `ready`
  - 文档 ready 后创建 run
  - 提交答案 -> 更新状态 -> 生成 settlement
  - 错题解释接口结合 `PageIndex + pgvector` 返回增强结果
  - 设置保存与读取一致
- **端到端测试**
  - 第一阶段主闭环：
    - 注册/登录
    - 上传文档
    - 等待文档处理完成
    - 进入任一模式
    - 完成一局
    - 查看结算
    - 查看错题解释
  - 第二阶段补充验证：
    - 赛季结算
    - Top 5 晋升
    - OAuth
    - 支付/订阅
- **验证重点**
  - 不只测成功路径，也要测：
    - 文档解析失败
    - PageIndex 不可用
    - LLM 超时
    - run 重复提交
    - refresh token 失效
    - 错题解释降级返回

**2. 风险与缓解**

- **风险：自部署 PageIndex 集成复杂度高**
  - 缓解：单独封装 `retrieval_backend` 适配层；先做健康检查、最小调用链路和本地 smoke test
- **风险：大模型输出不稳定，影响题库质量**
  - 缓解：所有题目生成使用结构化输出 schema；入库前加校验与最小清洗
- **风险：上传后处理链过长，影响首轮体验**
  - 缓解：采用后台任务；前端用 `job status` 轮询；状态明确区分 `processing / ready / failed`
- **风险：三种模式逻辑分叉后难维护**
  - 缓解：统一 `run` 模型，只把差异放到规则层和组题层
- **风险：开源版 / 闭源版后续分叉**
  - 缓解：从一开始抽象 `provider_adapter` 和 `model_registry`，业务层不直接耦合某家模型
- **风险：第二阶段赛季系统返工**
  - 缓解：第一阶段先预留周赛季和段位晋升相关表结构/服务边界
- **风险：pgvector 数据增长与索引维护成本**
  - 缓解：第一阶段仅向量化错题与复习样本，不把整套文档主检索压到 pgvector 上

**3. 约束与设计结论**

- 第一阶段主检索以 **PageIndex** 为核心，`pgvector` 只做错题增强
- 第一阶段 AI 先固定单模型运行，但后端必须具备未来多模型扩展能力
- 认证第一阶段只做邮箱密码，OAuth 第二阶段补齐
- 赛季与运营能力属于第二阶段，但第一阶段的数据模型必须兼容
- 前端当前无真实 API 包袱，后端可以按领域做干净接口

**4. 成功标准**

- [ ] 用户可以完成注册、登录、refresh、退出
- [ ] 用户可以上传文档并看到处理状态
- [ ] 文档处理完成后可生成基础题库并进入可学习状态
- [ ] 三种游戏模式都可基于统一题库创建 run 并完成答题
- [ ] run 结束后可返回标准 settlement
- [ ] 错题可以被记录，并可生成增强解释
- [ ] 基础问答和学习推荐可基于文档内容工作
- [ ] settings / profile 可持久化
- [ ] 第一阶段闭环可完成联调测试
- [ ] 第二阶段可在不推翻第一阶段架构的前提下补齐 OAuth、支付、赛季、运营后台
- [ ] 周赛季规则被明确纳入 V1 设计：周一至周日、Top 5 晋升下一段位
