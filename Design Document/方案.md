# 实施计划：文档驱动学习路径（方案 B）

## 概述
在现有 `library/home -> game-modes -> level-path -> learn` 链路上，新增“按文档+模式生成可版本化学习路径”的能力，并实现懒生成、失败重试、进度隔离、奖励风控和多格式统一解析。  
研发按阶段增量落地，用户侧一次性发布（待 Markdown + PDF/DOCX/PPTX 全部通过验收）。

## 需求
- 支持每文档每模式独立路径（endless/speed/draft）
- 路径结构：技能树（单元）→ 关卡（level=run）
- 首次进入 mode 懒生成；默认展示最新 READY 版本
- 路径版本并存，进度按 `path_version_id` 隔离
- 失败可重试，120s 超时提示
- 同文档同 mode 24h 最多 3 次重生
- 传奇复习奖励递减：50%→30%→20%→10%
- 免费用户每日封顶（UTC+8）：XP 300 / coin 60；订阅用户不封顶
- 旧版本可学，奖励统一 70%
- READY 永久保留，FAILED 7 天清理
- 解析管线统一：Markdown + MinerU（PDF/DOCX/PPTX）→ 标准化 Markdown

## 实施步骤

### 阶段 1：数据层与状态机落地（高优先）
1. **新增学习路径核心表**（文件: `backend/app/db/models/learning_paths.py`, `backend/alembic/versions/*`）
   - 操作：创建 `learning_path_versions / learning_path_nodes / learning_path_progress / legend_review_progress / path_regeneration_records`
   - 原因：把“硬编码 path options”升级为可持久化、可版本化结构
   - 依赖：无
   - 风险：高（迁移与索引设计）
2. **新增订阅与每日封顶表**（文件: `backend/app/db/models/subscriptions.py`, `backend/app/db/models/economy.py`）
   - 操作：新增 `subscriptions`, `daily_reward_cap_usage`
   - 原因：实现“仅订阅用户不封顶”的奖励规则
   - 依赖：步骤1
   - 风险：中
3. **扩展 run 与路径绑定字段**（文件: `backend/app/db/models/runs.py`, migration）
   - 操作：`runs` 增 `path_version_id`, `path_node_id`（保留 path_id 兼容期）
   - 原因：确保结算可追溯到路径节点
   - 依赖：步骤1
   - 风险：中

### 阶段 2：后端路径域服务 + 懒生成
4. **实现 LearningPathService**（文件: `backend/app/services/learning_paths/*.py`）
   - 操作：实现 `get_or_enqueue_latest_path()`、`regenerate_path()`、`list_path_nodes()`
   - 原因：统一处理懒生成、版本选择、重生频控
   - 依赖：阶段1
   - 风险：高
5. **改造 runs path-options 与重生接口**（文件: `backend/app/api/v1/runs.py`）
   - 操作：
     - `GET /api/v1/runs/path-options` 返回 `generation_status + latest_ready_version + nodes`
     - `POST /api/v1/runs/path-regenerations` 触发重生并做 24h/3 次限制
   - 原因：前端只需一个入口即可拿到“可展示+可轮询”状态
   - 依赖：步骤4
   - 风险：中
6. **worker 增加 path_generation 任务**（文件: `backend/app/workers/main.py` + 新 handler）
   - 操作：注册 `learning_path_generation` job，产出节点快照并写 version 状态
   - 原因：避免 API 阻塞，支持进度态与失败重试
   - 依赖：步骤4
   - 风险：高

### 阶段 3：run/结算/进度联动
7. **run 创建绑定路径节点**（文件: `backend/app/services/runs/service.py`, `backend/app/repositories/run_repository.py`, `backend/app/api/v1/runs.py`）
   - 操作：create_run 接收 `path_version_id/path_node_id`；按节点选题策略组卷
   - 原因：实现“点击某个 level 即学习该 level”
   - 依赖：阶段2
   - 风险：高
8. **结算规则接入奖励风控**（文件: `backend/app/services/learning_paths/reward_policy.py`, `backend/app/services/runs/service.py`）
   - 操作：接入传奇递减、旧版本70%、免费封顶、订阅实时判定、UTC+8 日切
   - 原因：经济规则统一在服务端闭环
   - 依赖：步骤7
   - 风险：高

## 风险与缓解
- **风险**：路径生成质量不稳定  
  - 缓解：严格 schema 校验 + fallback 最小路径（1 路径/2 单元）
- **风险**：120s 超时率高  
  - 缓解：异步 job + 前端轮询 + 超时后可继续拉取
- **风险**：奖励规则复杂导致误发  
  - 缓解：统一 RewardPolicy + 幂等键 + 审计日志
- **风险**：多格式解析失败率高  
  - 缓解：格式样本回归集 + 错误码分层 + MinerU降级策略
- **风险**：版本并存进度串线  
  - 缓解：所有进度写入必须包含 `path_version_id`
