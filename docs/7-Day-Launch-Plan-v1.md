# Xirang 7 天作战单 - v1

## 1. 本轮判断

- 当前阶段: 前端就绪, 后端与产品闭环待落地
- 核心问题: 不能再把时间花在前端 polish 上, 本周要把前端壳变成可验证的产品
- 推荐方案: 7 天只做“共享题库、共享结算、账本、购币、埋点、封测”

## 2. 本周唯一目标

- 验证激活率 >= 40%

## 3. 现在不做什么

- 不扩第四种模式
- 不上线 AI 额度兑换
- 不继续大幅重构前端视觉
- 不上播客/TTS/重特效
- 不做大而全管理后台

## 4. 7 天作战单

### Day 1 - 冻结规则与代码真相

- 动作:
  - 冻结本轮 PRD
  - 对齐前端现状与产品口径差异
  - 定义共享题库 schema / 共享结算接口 / 账本接口
- 输入:
  - `docs/PRD-Global-MVP-v1.md`
  - `frontend/src/pages/*`
- 输出:
  - 接口合同
  - 前端偏差清单
  - 共享状态机定义
- 验收:
  - 所有人对模式、结算、连胜、购币逻辑无歧义

### Day 2 - 打通文档解析与题库生成

- 动作:
  - 实现上传接口
  - 实现统一题库 JSON 返回
  - 加入解析失败降级到纯文本模板逻辑
- 输出:
  - 上传 -> 题库 JSON 可跑通
- 验收:
  - 任意样本资料能返回可消费题库
  - 失败能优雅降级

### Day 3 - 打通三模式共享引擎

- 动作:
  - 让三模式接同一题库
  - 去掉纯占位题目
  - 接入共享结算
- 输出:
  - 三模式能完成真实一局
- 验收:
  - `SpeedSurvival / EndlessAbyss / KnowledgeDraft` 都写入相同账本体系

### Day 4 - 账本、连胜与商城

- 动作:
  - XP / 代币账本落地
  - 连胜激冻库存与自动补充逻辑落地
  - 商城购买逻辑接入真实余额
- 输出:
  - 真实消费链路
- 验收:
  - 余额变化正确
  - 连胜激冻库存规则可复现

### Day 5 - 购币与风控

- 动作:
  - 接入最小购币包 $1.99
  - 商城常驻购币入口
  - 余额不足弹层
  - <2 秒不计分与收益递减规则写入
- 输出:
  - 最小购币闭环
- 验收:
  - 道具购买失败时能顺滑跳转购币
  - 过快答题不会产生错误奖励

### Day 6 - 埋点与小流量封测

- 动作:
  - 打激活漏斗埋点
  - 打题目错误反馈埋点
  - 找 5-10 位种子用户试玩
  - 收集每人首个失败点
- 输出:
  - 首轮行为数据
  - 首轮错误样本集
- 验收:
  - 明确前 3 个流失点

### Day 7 - 修复与 go / no-go

- 动作:
  - 只修 Top 3 阻塞问题
  - 复盘激活率、解析失败率、题目标错率、购币转化
  - 做 go / no-go 决策
- 输出:
  - 下周迭代清单
  - 发布门槛结论
- 验收:
  - 是否具备进入更大范围测试的条件

## 5. 指标看板

### North Star

- 每周有效学习关卡完成数

### Leading Indicators

- 激活率 >= 40%
- 上传完成率 >= 80%
- 上传后开局率 >= 70%
- 开局完成率 >= 75%
- 题目准确率 >= 80%

### Risk Metrics

- 解析失败率
- 用户标错率
- <2 秒作答比例
- 余额不足购币触发率
- 连胜激冻消耗率

## 6. 当前最大错配

- 前端已经很好, 但仍有“实现口径”与“产品口径”不一致
- 后端和评估闭环是本周真正的关键路径
- 若本周继续加模式细节或视觉 polish, 会牺牲激活验证速度

## 7. 决策门槛

### 继续条件

- 激活率 >= 40%
- 题目准确率 >= 80%
- 首批测试用户能完整跑通上传 -> 开局 -> 完局 -> 结算

### 切换条件

- 若开局率低, 优先修上传和模式选择页
- 若完局率低, 优先修题目质量和结算反馈
- 若解析失败高, 优先收窄输入与降级策略

### 停止条件

- 若 7 天后仍无法稳定跑通共享题库 + 共享结算, 暂停扩功能
- 若代币体系引发异常刷分占比 > 5%, 暂停购币实验

## 8. 本周最小行动包

1. 冻结接口与规则, 不再加功能
2. 打通统一题库和统一结算引擎
3. 用真实用户和真实数据决定下周该继续还是收缩
