# 排行榜功能优化设计规格

## 1. 背景与目标

### 1.1 问题描述
当前排行榜功能存在以下问题：
1. **用户名显示错误**：显示"Scholar 3ca3ca"而不是用户名"kabuto"
2. **载入更多按钮无效**：点击"加载更多学者"按钮没有反应
3. **每日专注内容硬编码**：显示"阅读《孙子兵法》"而不是用户实际学习内容
4. **好友榜功能未实现**：前端有切换按钮但后端没有实现
5. **数据更新频率**：需要30分钟更新一次排行榜数据

### 1.2 目标
优化排行榜功能，实现：
1. 显示正确的用户名（优先显示用户设置的显示名，其次显示用户名）
2. 修复载入更多按钮功能
3. 动态生成每日专注内容（使用LLM生成文档标题）
4. 实现好友榜功能（后续阶段）
5. 确保排行榜数据30分钟更新一次

## 2. 技术方案

### 2.1 选定方案：渐进式优化方案
基于现有代码结构进行优化，而不是全面重构。理由：
- 现有代码结构已经比较完整
- 只需要修复特定问题，不需要大规模重构
- 风险较低，可以逐步验证

### 2.2 关键改进点
1. **用户名显示优化**：确保前端正确显示用户名
2. **载入更多按钮修复**：检查并修复前端分页逻辑
3. **动态内容生成**：优化后端LLM标题生成逻辑
4. **好友榜功能**：后续实现好友榜数据过滤
5. **更新频率优化**：确保30分钟更新机制正常工作

## 3. 架构设计

### 3.1 数据流
```
用户学习文档 → Run记录 → Settlement XP → 排行榜API → 前端页面
```

### 3.2 组件关系
```
DungeonScholarLeaderboardPage.vue
├── LeaderboardSummaryPanel.vue (用户摘要)
├── LeaderboardStandingsTable.vue (排行榜表格)
└── API调用 → leaderboard.ts → 后端API
```

## 4. 接口设计

### 4.1 前端接口
- `getLeaderboardSnapshot(limit, offset, scope)` - 获取排行榜快照
- 保持现有接口不变，优化内部实现

### 4.2 后端接口
- `GET /api/v1/leaderboard` - 获取排行榜数据
- 保持现有接口不变，优化数据查询逻辑

## 5. 错误处理

### 5.1 用户名显示错误处理
- 当 `display_name` 为 null 时，回退到 `username`
- 当两者都为空时，显示"默认用户"

### 5.2 载入更多按钮错误处理
- 添加加载状态指示
- 处理网络错误和空数据情况

### 5.3 LLM标题生成错误处理
- 当LLM服务不可用时，回退到文档标题清洗
- 添加重试机制和超时处理

## 6. 测试策略

### 6.1 单元测试
- 测试用户名显示逻辑
- 测试载入更多按钮功能
- 测试LLM标题生成回退逻辑

### 6.2 集成测试
- 测试完整数据流
- 测试30分钟更新机制
- 测试错误处理逻辑

## 7. 风险与约束

### 7.1 风险
1. **LLM服务依赖**：需要确保LLM API可用
2. **数据库性能**：排行榜查询可能影响数据库性能
3. **前端兼容性**：需要确保不同浏览器兼容性

### 7.2 约束
1. **保持现有技术栈**：Vue 3 + FastAPI + PostgreSQL
2. **30分钟更新频率**：需要平衡实时性和性能
3. **用户体验优化**：需要确保界面响应迅速

## 8. 验收标准

### 8.1 功能完整性
- [ ] 排行榜显示正确的用户名（不是"Scholar 3ca3ca"）
- [ ] "加载更多学者"按钮正常工作
- [ ] 每日专注显示用户实际学习内容
- [ ] 排行榜数据30分钟更新一次

### 8.2 性能达标
- [ ] 排行榜页面加载时间 < 2秒
- [ ] 载入更多响应时间 < 1秒
- [ ] 30分钟更新机制正常工作

### 8.3 用户体验优化
- [ ] 界面响应迅速
- [ ] 错误处理友好
- [ ] 加载状态清晰

### 8.4 数据准确性
- [ ] 排行榜数据与数据库完全匹配
- [ ] 用户名显示正确
- [ ] 每日专注内容准确
